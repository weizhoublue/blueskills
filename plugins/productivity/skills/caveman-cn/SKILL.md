## 调用场景

**当用户没有明确要求调用本 skill 时，禁止使用***

**适用**
- 用户明确要求简短中文输出
- 用户提交中文正文，明确要求进行简短改写

**不适用**
- 用户未明确要求使用本 skill

## 双模式

| 模式 | 判定 | 输出 |
|------|------|------|
| 改写 | 用户提交待处理正文 | 仅压缩后全文，零说明 |
| 风格 | 无待改写正文 | 当次回复全文按规则写 |

**持续：仅当次回复**

## 压缩原则

### 1. 删低信息量词

默认删除不影响原意的低信息量词。

包括不限于：
- 相关、具体、详细、完整、主要、整体
- 一些、很多、比较、相对
- 其实、显然、可以看到、我们发现
- 所谓的、对应的、这个、那个、这种、那种
- 其实/真的/基本/当然/当然可以/我很乐意/觉得/可能/似乎/请/您/麻烦/感谢/啊/吧/呢/哦/啦/什么/认为/简直

必要时可保留，例如技术固定表达、用户原意依赖该词。

### 2. 删口语和语气

默认删除不影响事实含义的口语表达。

包括不限于：
- 我觉得
- 感觉
- 就是说
- 换句话说
- 怎么说呢
- 从我的角度看
- 这里想表达的是
- 个人判断

不得删除表达不确定性的词，例如：可能、疑似、大概率、看起来。除非上下文已证明结论成立。

### 3. 长表达改短表达

将冗长表达压成短词。

包括不限于：
- 进行检查 → 检查
- 根据 → 据
- 如果 → 若
- 产生影响 → 影响
- 没有 → 未
- 无法正常工作 → 不可用
- 和当前变更直接相关 → 与变更相关

### 4. 合并重复

同一事实、原因、条件、后果、结论只表达一次。

保留优先级：
1. 关键条件
2. 直接原因
3. 明确结论
4. 用户可见后果
5. 最短修复动作

### 5. 字段短标签化

字段名只做导航，不做解释。

要求：
- 优先 2～4 字
- 最多 6 字
- 使用名词或动宾短语

示例：
- 相关代码证据 → 证据
- 缺陷的触发条件 → 触发
- 造成的代码后果和业务功能后果 → 后果
- 建议解决方案 → 修复
- 反证检查 → 对照
- 完整调用链说明 → 调用链
- 防护未生效原因 → 原因

### 6. 句式压缩

将解释性表达压成判断句，优先使用“条件 + 结果”

示例：
- 这里的问题在于代码没有处理配置缺失的情况 → 代码漏处理配置缺失场景

## 保真

**不改变文本的组织格式，例如列表、引用、代码块、markdown、jsonl 等**

**必须保留**
- 原意
- 主体
- 条件
- 因果
- 结论
- 限制
- 风险
- 关键细节

禁止：
- 压缩成总结
- 删除关键条件
- 改变因果关系
- 改变确定性程度
- 把“可能”改成“必然”
- 把局部问题改成全局问题
- 引入原文没有的新结论

## 保护范围

零改动：
- fenced code block
- 反引号内文本
- 代码、函数名、路径、配置键、API 名、URL
- 栈信息、报错原文、日志原文

混合中英文时，只压缩中文叙述。

## 边界

- 改写模式代码为主：只压代码外中文
- 丢条件/结论：补回后再输出
- 说明性引用可压；技术标识不动
