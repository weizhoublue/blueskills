  - [`2026-06-04-audit-code-report-quality-design.md`](2026-06-04-audit-code-report-quality-design.md)（历史 `trigger.scenario` 与含糊拒收）
  - [`2026-06-03-audit-pr-plugin-design.md`](2026-06-03-audit-pr-plugin-design.md)（`trigger_vague_unfounded` / 生产入口）
- 背景：`plugins/audit/skills/review/SKILL.md` 要求「明确触发条件」，但未约束层级与具象程度。agent 常将函数内部分支（`ptr==nil`、某 guard 未走）写成触发条件，或用「可能」「大概」「未来」等含糊语填充，缺少顶层 config/input 与可评估的具体取值。

## 1. 问题陈述

当前 `缺陷的触发条件` 字段典型不足：

1. **层级偏低** — 触发条件写为函数内部状态/分支，而非用户/运维可见的 config、API、CLI、env、CR spec
2. **无数值** — 缺少超时秒数、副本数、重试次数、枚举字面量等可核对取值
3. **含糊措辞** — 「可能」「大概」「未来」「某些情况下」出现在参考场景或触发条件中
4. **职责混淆** — 调用链、缺陷落点分支与顶层触发前提混写在同一字段

目标：在保留现有 finding 自包含结构与四阶段流程的前提下，将触发条件**场景化、量化、顶层化**；函数级路径归入**可达性证据**。

## 2. Brainstorming 决策摘要

| 决策点 | 选择 |
| --- | --- |
| 顶层严格度 | **B — 分层表达**（对齐 investigate R21） |
| 落点 | **finding 字段内双块**；不新增报告 `###` 子节 |
| 规则命名 | **T1/T2/T3**（audit 专用，与 investigate R17/R20/R21 正交对应） |
| 参考场景含糊语 | **严禁**「大概」「可能」「未来」及同义 hedge；须具体值 |
| 无法量化时 | 量化观测字段写「未能从代码量化」；**禁止**用含糊句替代；若整条参考场景只能含糊表述 → 质检按证据不足**删除** finding |
| 明确不做 | 终稿独立触发条件附录；编造 PR/代码无法佐证的生产流量数值 |

## 3. 因果模型：T1/T2/T3

| 规则 | 说明 |
| --- | --- |
| **T1 顶层逻辑** | 触发条件的**顶层逻辑条件**须从用户/运维可见入口表述：config / input / env / API / CLI / CR spec；**禁止**仅用函数内变量、局部分支、指针状态作为逻辑条件 |
| **T2 场景证据** | 逻辑条件须有代码或 PR 依据（`path:line` 或 PR 原文）；hedge 语不得进入逻辑条件列表 |
| **T3 场景具象化** | 逻辑条件之后须提供 **参考触发场景** 块（≥1 条 vignette）；须含具体配置取值、业务输入参数、量化观测；**严禁**「大概」「可能」「未来」「某些情况下」等 |

**分层边界（硬约束）：**

1. **顶层逻辑条件** — 抽象布尔/枚举前提（config / input / runtime 部署态），须 refs
2. **参考触发场景** — 同字段内、逻辑条件之后的具象块；必须具体值；禁止 hedge
3. **可达性证据**（既有 `相关代码证据` 子项）— 调用链、函数级落点、guard 未挡住的原因

## 4. `缺陷的触发条件` 字段结构

### 4.1 模板（候选缺陷 + 终稿共用）

```markdown
- 缺陷的触发条件：
  - **顶层逻辑条件**（须同时满足；T1 + T2）：
    - 条件1（config/input）：... refs: path:line
    - 条件2（runtime/部署态）：... refs: path:line
  - **参考触发场景**（可评估；T3；不计入逻辑条件编号）：
    - **场景1**（来源：code_synth | pr_context | hybrid）
      - **映射条件**：条件1 + 条件2
      - **配置快照**：`key=value` … refs: path:line 或 PR 引用
      - **业务输入**：API/CLI/请求 + 关键参数与取值
      - **应用层行为**：用户可理解的操作结果（非函数名罗列）
      - **量化观测**：具体数字 + 单位；无法则写「未能从代码量化」
```

### 4.2 场景来源枚举（audit 专用）

| 值 | 含义 | 数值依据 |
| --- | --- | --- |
| `code_synth` | 从 diff、默认值、schema、常量合成的可触发示例 | 配置/阈值须 refs |
| `pr_context` | PR 标题/描述/comments 声称的触发情境 | 标 PR 原文；与代码不符须点明 |
| `hybrid` | PR 现象 + 代码补全缺失字段 | 分项标依据，禁止整段含糊 |

### 4.3 禁止词表（参考触发场景 + 逻辑条件）

**严禁**（含同义变体）：

- 大概、可能、也许、似乎、潜在、有一定概率、在某些情况下
- 未来、将来、若以后、一旦升级后（无当前 diff/代码依据时）
- 无量化形容：很高、很低、较大、较短、设得很小（无具体数字时）
- 无主体操作：用户执行某操作后、配置不当时

**最终报告**整体仍遵守既有禁止词（阶段 3 质检列表）。

### 4.4 内容归属表

| 内容 | 归属 |
| --- | --- |
| `spec.replicas=3`、`strategy: Recreate` | 顶层逻辑条件 / 配置快照 |
| `POST /api/v1/foo` body 字段取值 | 业务输入 |
| controller 删除旧 Pod | 应用层行为 |
| `handleFoo()` → `validateBar()` → nil deref | **可达性证据** |
| `if ptr == nil` 分支 | **可达性证据**（缺陷落点），**不是**顶层逻辑条件 |

## 5. 阶段 2 改造（2a/2b/2c/2d 共享）

### 5.1 共享规则新增 T1–T3

在「缺陷成立条件」之后、「缺陷性质」之前插入 **触发条件规则（T1–T3）** 三节条文（见 §3）。

### 5.2 字段要求 `### 缺陷的触发条件` 重写

替换现有简短说明为 §4.1 模板 + 禁止词表 + 归属表摘要。

### 5.3 各 scanner 追加工作步骤

在输出候选缺陷前：

1. 从 diff/锚点反向提取：相关配置键、默认值、schema 约束、常量（Grep/Read）
2. 从阶段 1「变更声称」提取 `pr_context` 素材（若有）
3. 合成 ≥1 条参考场景；映射到顶层逻辑条件编号
4. 函数级调用链**只**写入 `相关代码证据 → 可达性证据`，**不得**作为顶层逻辑条件
5. 参考场景自检：无禁止词；配置/输入/量化须具体值或「未能从代码量化」

### 5.4 候选缺陷输出格式

`缺陷的触发条件：` 行改为 expandable 结构（见 §4.1），替换原单行占位。

## 6. 阶段 3 质检改造

### 6.1 新增拒收项（证据不足 → 删除）

| 代号 | 说明 |
| --- | --- |
| `trigger_function_level_only` | 触发条件仅有函数内部分支/变量，无顶层 config/input |
| `trigger_scenario_hedge` | 参考触发场景含禁止词表用语 |
| `trigger_scenario_no_concrete_value` | 参考场景缺配置具体取值、或缺业务输入具体参数、或量化观测仅为含糊形容 |
| `trigger_logic_hedge` | 顶层逻辑条件含 hedge 且无 tier 标注 |

### 6.2 处理规则补充

- 成立：补全顶层逻辑条件 + 参考场景具体值，去除禁止词
- 不成立 / 证据不足：删除（与现规则一致）
- **禁止**仅润色含糊语而不补具体值
- 量化观测无法从代码得出：允许写「未能从代码量化」，**禁止**用「大概 N 秒」替代

### 6.3 与严重等级

- T 规则拒收**不单独**改变 P0–P2 定级矩阵；但含糊/无顶层触发的 finding 应在质检阶段删除，不进入终稿

## 7. 阶段 4 终稿

## 9. 正反例

### 9.1 好例子

```markdown
- 缺陷的触发条件：
  - **顶层逻辑条件**（须同时满足；T1 + T2）：
    - 条件1（config）：`spec.strategy=Recreate` refs: types.go:88
    - 条件2（runtime）：滚动更新触发旧 Pod 删除 refs: controller.go:210
  - **参考触发场景**（可评估；T3）：
    - **场景1**（来源：code_synth）
      - **映射条件**：条件1 + 条件2
      - **配置快照**：`replicas: 3`，`strategy: Recreate`，`terminationGracePeriodSeconds: 30` refs: types.go:42,88
      - **业务输入**：`kubectl apply` 更新 Deployment，仅修改 container image refs: PR description
      - **应用层行为**：controller 删除旧 RS 下 3 个 Pod，新 Pod 未 Ready refs: controller.go:210-225
      - **量化观测**：终止宽限 30s 内可用副本数降为 0 refs: types.go:42
```

### 9.2 坏例子（应删除或重写）

- 触发条件：`当 ptr 为 nil 时进入 handleFoo` — 函数级
- 配置快照：`replicas 较高时` — 无量化
- 业务输入：`用户未来升级后可能触发` — 禁止词 + 非当前依据
- 量化观测：`大概 30s 内可能不可用` — 禁止词
