# audit review 审计范围排除设计

**日期：** 2026-06-06  
**状态：** 已实施  
**目标文件：** `plugins/audit/skills/review/SKILL.md`

---

## 背景

`plugins/audit/skills/review/SKILL.md` 定义四阶段缺陷审计流程。当前仅在主编排合并处有「合理 skip 示例」提及 `docs/*.md`，未对依赖目录、测试代码、文档目录做系统性排除。审查 vendor 大 diff、测试变更和 docs 变更会浪费 token，且易引入与生产代码无关的噪音 finding。

同类插件 `plugins/investigate-issue/skills/investigate/SKILL.md` 已有 `vendor/`、`node_modules/`、`third_party/` 等排除先例，本设计在 review skill 上落地类似机制，并按 PR 审查场景收窄测试与文档范围。

---

## 目标

在 review skill 的**全部审计环节**（预处理、阶段 1～4）统一忽略以下变更：

1. 依赖目录：`vendor/`、`node_modules/`、`third_party/`
2. 测试代码：`*_test.go` 及独立 e2e 目录（`e2e/`、`test/e2e/`、`tests/e2e/`）
3. 文档：`docs/` 目录

当 PR 变更**全部**落在排除范围内时，快速结束（跳过阶段 2/3 缺陷扫描），输出 `REVIEW_RESULT=review_mark_ignore`。

---

## 已确认需求

| 维度 | 决策 |
|------|------|
| 测试排除 | `*_test.go` + 独立 e2e 目录（`e2e/`、`test/e2e/`、`tests/e2e/`）；**不含**通用 `test/`、`tests/`、`__tests__/`、`spec/` |
| 依赖排除 | `vendor/`、`node_modules/`、`third_party/` |
| 文档排除 | 仅 `docs/` 目录 |
| 全部为排除路径 | 快速结束：不委派阶段 1/2/3，直接输出无生产代码可审结论 + `review_mark_ignore` |
| 实现方案 | 主编排预过滤 diff（方案 A） |

---

## 方案选择

### 方案 A：主编排预过滤 diff（采用）

读取 diff 后、阶段 1 之前，按路径规则剥离排除文件 hunk，生成「可审 diff」；各阶段只接收可审 diff。可审文件为空 → 快速结束。

### 方案 B：共享排除规则，各阶段自行跳过（不采用）

改动小，但 agent 仍可能读到 vendor 大 diff，2d Grep 易扫到噪音，执行不一致。

### 方案 C：新增阶段 0 分类 agent（不采用）

多一轮委派，全排除时仍浪费 Task，过度设计。

---

## 设计

### 1. 审计范围排除（HARD-GATE）

在 SKILL.md「输入」之后、「总体流程」之前新增全局章节。

#### 排除路径（任一规则命中即整文件排除）

| 类别 | 路径规则 |
|------|----------|
| 依赖 | `vendor/**`、`node_modules/**`、`third_party/**` |
| 测试 | `**/*_test.go`、`e2e/**`、`test/e2e/**`、`tests/e2e/**` |
| 文档 | `docs/**` |

#### 匹配语义

- 以 diff 中的**文件路径**为准（正斜杠，相对仓库根）
- 前缀或 glob 命中即整文件排除，不拆 hunk
- 混合 PR：只保留未命中规则的文件进入「可审 diff」

#### 排除后果（全环节适用）

1. **不分析**排除文件中的变更
2. **不引用**排除文件作为缺陷证据
3. **Grep/Read 范围**：2b 上下游、2d 同类残留搜索时，不在排除路径内搜索或采信结果

---

### 2. 变更预处理（阶段 0，主编排，不委派）

置于「总体流程」之前，更新流程图为：

```text
0. 变更预处理（主编排）
1. 变更意图分析
2. 代码缺陷扫描（并行 2a / 2b / 2c [+ 条件 2d]）
3. 缺陷质检
4. 报告拼装
```

#### 主编排步骤

```text
1. 读取 changed files 列表与完整 diff
2. 按排除规则分为「可审」与「已排除」
3. 剥离已排除 hunks → 生成「可审 diff」
4. 可审文件列表为空 → 快速结束（不委派 1/2/3）
5. 否则 → 可审 diff + 排除摘要传入阶段 1 及后续
```

#### 排除摘要（编排上下文，不进入最终报告正文）

```markdown
## 变更预处理摘要
- 可审文件数：N
- 已排除文件数：M
- 已排除路径（按类别）：
  - 依赖：…（共 x 个文件）
  - 测试：…（共 y 个文件）
  - 文档：…（共 z 个文件）
```

#### 快速结束（可审文件数 = 0）

不委派 sub-agent，主编排直接输出：

```markdown
## 代码变更背景

本次变更仅涉及依赖目录（vendor/node_modules/third_party）、测试代码（*_test.go / e2e 目录）或文档（docs/），无生产代码变更纳入审计范围。

## 缺陷

未发现满足审计范围条件的代码缺陷（已排除依赖、测试与文档变更，不在审查范围内）。

## 最终结论

REVIEW_RESULT=review_mark_ignore
```

#### 混合 PR

- 阶段 1～4 仅使用「可审 diff」
- 阶段 1「涉及的主要文件」只列可审文件
- 最终报告背景段可一句说明：「已排除 N 个依赖/测试/文档文件，未纳入本次审计」

---

### 3. 各阶段行为调整

#### 阶段 1（变更意图分析）

- 输入：**可审 diff** + PR 元数据
- PR 标题/描述/comments 仍可读，但不得从已排除文件推断缺陷或扩展范围
- 变更性质以可审内容为准
- 「涉及的主要文件」不得包含已排除文件

#### 阶段 2（2a / 2b / 2c / 2d）

**共享规则新增（HARD-GATE）：**

> 传入的 diff 已由主编排预处理，所列文件均在审计范围内。编排上下文中的「已排除路径」不得作为缺陷证据或扫描锚点。Grep/Read 时不得深入已排除路径（2b 上下游、2d 全仓 Grep 均适用）。

委派 prompt 组装：

- 传入 **可审 diff**（非原始 diff）
- 附排除摘要（只读）

**2b 补充：**

- 上下游清单中，caller/callee 若在排除路径内 → 注明 `已排除（测试/依赖/文档），不核实`
- 不计入「应扫未扫」

**2d 补充：**

- Grep 默认排除 vendor/node_modules/third_party、`*_test.go`、e2e 目录、docs/
- 命中排除路径的匹配不进入残留候选清单

#### 阶段 3（质检）

- 候选缺陷证据若落在已排除路径 → 删除（防御性规则）
- 不重新审计已排除文件

#### 阶段 4（报告拼装）

- 混合 PR：背景段可选一句排除说明
- 全排除：使用快速结束模板，不走常规模板

---

### 4. 与现有条文的关系

- **删除/替换**主编排合并处「合理 skip 示例」中的 `纯 docs/*.md`（由全局排除规则统一承担）
- **保留**「不要把测试不足单独作为缺陷」
- **保留**「忽略 P3 级别缺陷」等既有规则

---

### 5. 实施范围

仅修改 `plugins/audit/skills/review/SKILL.md`。可选：将 `plugins/audit/.claude-plugin/plugin.json` 的 `description` 增补「排除依赖/测试/docs 变更」一句（非必须）。

---

## 不在范围内

- 不排除通用 `test/`、`tests/`、`__tests__/`、`spec/`（用户明确选择方案 B 级测试范围）
- 不排除根目录 `README.md`、`CHANGELOG.md` 等非 `docs/` 路径的文档
- 不修改 investigate-issue 或其他插件
- 不新增 agent 文件或自动化脚本

---

## 验收标准

1. SKILL.md 含完整的排除路径表、预处理步骤、快速结束模板
2. 阶段 1～4 及共享规则均引用「仅可审 diff」约束
3. 2b/2d 对排除路径的 Grep/清单 skip 行为有明文规定
4. 全排除 PR 不委派阶段 1/2/3，输出 `review_mark_ignore`
5. 混合 PR 仅审查可审文件，报告可注明排除统计
