# investigate 阶段 2a/2b 两阶段扫描与 2b 前提核实设计

**日期：** 2026-06-06  
**状态：** 待实施  
**插件：** `plugins/investigate-issue`（skill：`investigate`）

---

## 背景

`plugins/investigate-issue/skills/investigate/SKILL.md` 阶段 2 并行委派 **2a 代码追踪** 与 **2b 业务上下文分析**。实践中两类问题突出：

1. **2a**：C0–C4 调用链、触发条件、后果写在同一段「工作内容」里，发现与落地判断混在一起，agent 易按 `issue_brief` 编造中间 hop，或跳过 `path:line` 核实。
2. **2b**：B1–B5 默认用户描述的功能真实存在，缺少「用户主张是否在代码中成立」的硬门禁；复盘时常发现用户假设的能力**未实现或不存在**，但业务因果链已写成 `(confirmed)`。

本设计参考 audit review 2b 重构思路（清单 → 逐项核实），并针对 investigate 保留 C0–C4 与 B1–B5 交付物。

---

## 目标

1. **2a 可执行化**：两阶段发现 + 落地——入口与一层追踪清单 → C0–C4 骨架（仅引用清单）→ 逐格核实。
2. **2b 先验真、再讲透**：阶段 A 核实用户每条可核对主张；阶段 B 在功能层面完整解读问题（B1–B5 + 必填「功能层面完整解读」）。
3. **phantom 功能防护**：核心能力 `missing` / `doc_only` 时，禁止在 B2–B5 以 `(confirmed)` 描述该能力已上线。
4. **保持输出契约**：阶段 3 三节报告、R16–R20、阶段 4 深化、终稿格式不变；`issue_true` / `issue_false` 仍由主编排阶段 3 按 R19 判定。
5. **Read 预算**：阶段 1 保留现有 Read ≤40；**2a/2b 暂不删除 Read 上限**（与 audit 不同，待后续观测再议）。

- 不改变阶段 0/1/4/5 主流程与终稿模板结构。
- 不将 `issue_false` 判定下沉给 2b 独占（2b 提供前提依据，主编排综合 R19）。
- v1 仅改 `SKILL.md`（及可选 `plugin.json` 版本/描述）。

```text
阶段 1 问题信息搜集
  → 并行 2a 代码追踪 + 2b 业务上下文分析
```

阶段 2 仍为**两次并行** `Task`；变化在各自**内部两阶段**与强制清单输出。

## 2a — 代码追踪（重写工作流程）

**角色：** 代码追踪员  
**聚焦：** 函数级调用链 C0–C4（每步 `confirmed` + `path:line` + 业务含义）。

### 必扫流程（须按序执行）

#### 阶段 A1 — 入口与一层调用清单（只建表）

1. 从阶段 1「候选模块」「入口线索」确定 C0 **候选入口**（config/env/api/cli）。
2. 对每个入口在同仓库内找**直接下一跳**（`Grep` + 必要时 `Read` 调用点 / 入口函数体）。
3. 合并去重，在输出中必填 **追踪清单**：

```markdown
### 追踪清单

| ID | 方向 | 符号 | 文件 | 关联入口 | 发现方式 |
|----|------|------|------|----------|----------|
| T1 | 入口 | handleX | pkg/a.go | C0候选 | read/grep |
| T2 | 下一跳 | dispatchY | pkg/b.go | T1 | read body |
```

- 无下一跳：行内注明 `leaf / inline-only`。
- **阶段 A1 禁止**：缺陷判断、无依据的 C0–C4 叙事、无清单项锚定的深搜。

#### 阶段 A2 — C0–C4 骨架（仅引用清单）

仅用追踪清单行拼出 C0–C4 骨架；每行须带 `清单ID`；清单无对应 hop → 标 `gap`，**禁止**编造符号：

```markdown
### 调用链骨架（待核实）

| 层 | 清单ID | 候选符号 | 业务含义（待核实） |
|----|--------|----------|-------------------|
| C0 | T1 | handleX | ... |
| C1 | T2 | dispatchY | ... |
| gap | — | — | 清单无下一跳；阶段 B 扩层或标未能确认 |
```

#### 阶段 B — 逐格落地核实

- 对骨架每一行（含 `gap`）Read 填 `path:line`、业务含义、证据层级 `(confirmed)`。
- **C2** 允许 guard/分支等非 callee 行，但须 `path:line`。
- **按需扩展**：核实中发现具体疑点（如参数透传、未处理错误）时，可向下一层扩一层；须在输出记录：**扩展项 | 疑点 | 扩到哪 | 结论**。
- 骨架 `gap` 未在阶段 B 解决 → 该 hop **不得**标 `(confirmed)`，写入 **未能确认的主张**。
- 产出既有字段：入口点、函数级调用链（C0–C4）、缺陷落点、触发条件（R17）、后果、未能确认的主张。

**不做：** 业务前提是否属实（2b）；修改源文件。

**覆盖说明门禁（主编排可选检查，写入 2a 输出末尾「扫描覆盖说明」）：**

- `- [ ] 追踪清单已覆盖阶段 1 全部候选入口`
- `- [ ] C0–C4 骨架每行有清单ID或 gap 标记`
- `- [ ] 骨架每一行已核实或记入未能确认的主张`
- `- [ ] 扩展项（若有）已记录`

## 2b — 业务上下文分析（重写工作流程）

**角色：** 业务上下文分析员  
**聚焦：** ① 用户主张是否属实；② 功能层面完整解读问题。

### 必扫流程（须按序执行）

#### 阶段 A — 前提核实（只建表，不写 B1–B5 因果结论）

从 `issue_brief` + 阶段 1 输出拆解每条**可核对主张**（功能、接口、配置行为、用户可见现象等），在代码/文档中核实：

```markdown
### 前提核实

| ID | 用户主张 | 核实结果 | 证据 | 说明 |
|----|----------|----------|------|------|
| P1 | 「支持功能 X」 | exists | path:line | 有完整实现路径 |
| P2 | 「Y 会报错」 | partial | path:line | 仅部分路径实现 |
| P3 | 「Z 接口可用」 | missing | 搜索：... | 无实现或仅 stub |
| P4 | 「文档称支持 W」 | doc_only | docs/... | 代码无对应 |
```

**核实结果枚举：**

| 结果 | 含义 |
|------|------|
| `exists` | 代码有完整或主路径实现 |
| `partial` | 有入口/flag/API，但主路径未实现、stub、或默认关闭 |
| `missing` | 代码中找不到对应能力 |
| `doc_only` | 仅文档/注释声称，代码无实现 |

**阶段 A 禁止**：写 B1–B5 因果结论；将用户原话标为 `(confirmed)`。

#### 阶段 B — 功能层完整分析

在前提表基础上输出：

**1. 业务因果 B1–B5**（须引用 P* ID；证据层级遵守全局规则）：

| 层 | 内容 | 约束 |
|----|------|------|
| B1 情境 | 谁、部署/配置、**代码实际能力边界** | 用户声称与代码不符须点明 |
| B2 坏结果 | 用户可观察坏结果 | 核心前提 `missing`/`doc_only` → 不得 `(confirmed)`；须区分「实际」vs「若前提成立则…」`(inference)` |
| B3 兄弟路径 | 默认/兄弟为何不同 | 必填；无 peer 须说明原因 |
| B4 介入点 | 缺陷在业务流哪一段 | 须对齐 2a C3 或说明无法对齐原因 |
| B5 影响 | 功能/性能/可靠性 | 区分实际影响 vs 假设前提下的影响 |

**2. 必填 `### 功能层面完整解读`**

2–4 段叙事：该功能**业务上应做什么**、**代码实际怎么做**、**二者差异**；禁止以 path 清单代替叙事（R16）。

**3. 保留项**：兄弟分支对比、不触发场景、关键机制动机 W1–W3（可选，规则不变）。

**规则摘要：**

- `issue_brief` 每条**核心主张**须在阶段 A 有 P* 行。
- 核心能力 `missing` / `doc_only`：**禁止**在 B2–B5 用 `(confirmed)` 描述该能力已上线。
- `partial`：必须写清「已实现哪段 / 缺哪段」。
- `inference` 须有 uncertainty_note（含「未能从代码确认」）。

**与 2a 衔接：**

- 2a 骨架 `gap` 与 2b 前提 `missing` 指向同一能力 → 阶段 3 写入 `### 未能从代码确认的前提`。
- 2b B4 不得描述 2a 未 `(confirmed)` 的链路 hop。

**覆盖说明门禁：**

- `- [ ] 前提核实表已覆盖 issue_brief 核心主张`
- `- [ ] 功能层面完整解读已填写`
- `- [ ] B1–B5 已执行或注明 skip（及原因）`
- `- [ ] 无 phantom 功能被标为 confirmed`

## 阶段 3 主编排（微调）

综合步骤 1 增加显式顺序：

1. **先读** 2b `### 前提核实` 与 2a `gap` / **未能确认的主张**。
2. 若核心前提 `missing`/`doc_only`：在初稿 `### 未能从代码确认的前提` 中列出；B2–B5 / C 链不得升格为 confirmed。
3. `issue_true` / `issue_false` 仍按既有 R19 规则；`issue_false` 典型情形补充：**无法 confirmed 用户前提**（功能不存在/未实现）。

---

## 阶段 4 深化（微调）

评审维度可增补（major，非单独 blocking）：

- **phantom 前提**：正文或 2b 将 `missing` 能力写作 confirmed 业务链。
- **rollback**：若 call_chain blocking ≥2 且伴随前提 missing，可优先重委派 **2b**（而不只 2a）。

## 验收标准

1. 2a 输出含 **追踪清单**、**调用链骨架**、核实后 C0–C4（含 gap 处理规则）。
2. 2b 输出含 **前提核实**表、**功能层面完整解读**、B1–B5。
3. 核心能力 `missing` 时，B2–B5 无 confirmed 「已上线」表述。
4. 阶段 3/4/5 终稿格式与 R16–R20 不变。
5. 阶段 1 Read ≤40 保留；2a/2b 未引入新的 Read 上限删除（本版 scope）。

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 清单+骨架 token 增加 | 强制两阶段防止返工编链；gap 早暴露 |
| 2a/2b 并行信息不一致 | 阶段 3 先合并前提与 gap；阶段 4 可 rollback 2b |
| partial 判不准 | 枚举定义 + 须写「已实现/未实现」分段 |
| C2 非 callee 分支漏扫 | 阶段 B 明确允许 guard 行且须 path:line |
