---

## 1. 背景与目标

用户需要约束大语言模型中文输出的表达方式：更短、更直接，去掉低信息量词和口语，但不丢原意。

**目标：**

- 新建 `productivity` 插件，v1 仅含 `caveman-chinese` skill
- 双用途：当次回复风格约束 + 显式文本改写
- **skill 自身须简短**：正文践行压缩原则，避免 skill 描述比压缩效果还冗长

**非目标（v1）：**

- 不新增 sub-agent、校验脚本、`examples.md`
- 不压英文叙述（英文技术术语原样保留）
- 不做会话级持久风格（仅当次回复）

## 3. 双模式与触发

### 3.1 双模式（同一次回复内判定）

| 模式 | 判定 | 输出 |
|------|------|------|
| **改写** | 用户提交待处理正文（粘贴、附件、引号块） | 仅输出压缩后全文，零说明 |
| **风格** | 无待改写正文，用户要简短回复 | 当次回复全文按规则写 |

### 3.2 触发

- **显式：** `/productivity:caveman-chinese`、attach skill
- **关键词：** `压缩`、`精简`、`简短`、`去冗余`、`caveman`（写入 frontmatter `description` 与正文各 1 行）

### 3.3 持续范围

仅当次回复；下条消息恢复默认表达。

### 3.4 引用处理

说明性中文可压；路径、栈、报错原文、代码标识符不动。

## 4. 表达原则

写入 `SKILL.md` 为紧凑表，每行一条：

| # | 规则 | 操作 |
|---|------|------|
| 1 | 禁低信息量词 | 删：相关、具体、详细、完整、主要、整体、一些、很多、比较、其实、显然、可以看到、我们发现… |
| 2 | 禁口语语气 | 删：我觉得、感觉、就是说、换句话说、怎么说呢、从我的角度看、这里想表达的是… |
| 3 | 长短语改短 | 进行检查→检查；进行处理→处理；产生影响→影响 |
| 4 | 去重复 | 同一事实/原因/后果/结论只写一次 |
| 5 | 句式压缩 | 解释句→判断句；长条件句→「条件 + 结果」 |

### 4.1 保真（HARD-GATE）

- 保留原意、条件、因果、结论、关键细节
- 压缩 ≠ 总结（禁止删条件、合并不同结论）
- 代码字符、函数名、路径、配置键、API 名、行内/fenced 代码块 — 零改动

### 4.2 压缩范围

| 压 | 不压 |
|----|------|
| 段落、标题、列表项、表格单元格中的中文 | fenced code block、反引号内、URL、文件路径、栈追踪、报错原文 |

### 4.3 内嵌样例（≤5 组，表格式）

| 前 | 后 |
|----|-----|
| 它的主要作用就是对用户输入进行检查 | 检查用户输入 |
| 我们可以看到，当配置缺失时会产生影响 | 配置缺失时功能失效 |
| 其实这个问题主要是因为没有进行处理 | 未处理导致此问题 |

## 5. SKILL.md 篇幅预算

| 区块 | 行数上限 |
|------|----------|
| frontmatter | 3 行 |
| meta 规则（本文件须践行下文原则） | 2 行 |
| 调用场景 / 双模式 | 各 ≤8 行 |
| 5 条表达原则 | ≤15 行（表） |
| 保真 + 禁区 | ≤10 行 |
| 输出前自检 | 3 条 bullet |
| before/after 样例 | ≤8 行 |

**硬上限：** ≤80 行 / ~1200 字。

---

## 6. 边界与错误处理

| 场景 | 处理 |
|------|------|
| 用户只触发关键词、无正文 | 风格模式：当次回复全文压缩 |
| 混合中英文 | 只压中文叙述；英文术语保留 |
| 改写模式收到代码为主内容 | 只压代码外中文注释/说明；代码本体不动 |
| 压缩后丢条件/结论 | 禁止输出；补回后再给 |
| 用户未触发 | 不应用本 skill |

**输出前自检（内嵌 skill 末尾，3 条）：**

1. 禁词表 + 口语表扫一遍
2. 条件、因果、结论是否齐全
3. 代码块、路径、标识符是否原样
