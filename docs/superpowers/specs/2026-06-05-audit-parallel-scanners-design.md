# audit 阶段 2 并行 scanner 拆分设计

**日期：** 2026-06-05  
**状态：** 已实施  
**插件：** `plugins/audit`（skill：`review`）

---

## 背景

当前 `plugins/audit/skills/review/SKILL.md` 的阶段 2「代码缺陷扫描」将 **9 个审计维度** 交给 **单个** sub-agent 执行。实践中以 **漏报（假阴性）** 为主：维度过多、上下文拥挤，agent 容易扫不全上下游、边界与兼容性。

用户提议将阶段 2 拆为三个 specialist agent，并行扫描后由主编排合并，再进入既有质检与报告拼装。

---

## 目标

1. **降低漏报**：每个 agent 职责单一、带显式 checklist 与「扫描覆盖说明」。
2. **保持输出契约**：候选缺陷格式、缺陷成立条件、P0–P2 等级、最终报告与 `REVIEW_RESULT` **不变**。
3. **保持通信方式**：Markdown 在 prompt 内传递；无 JSON、无临时文件、无 shell 脚本。
4. **v1 实现范围**：仅修改 `SKILL.md` + `plugin.json` 版本/描述；不新增 `agents/*.md` 文件。

---

## 非目标

- 不增加独立 merger sub-agent（主编排合并）。
- 不默认第二轮「补扫」agent（记为后续可选增强）。
- 不改为 `audit-code` 式 JSON 中间态。
- 不修改阶段 1 / 阶段 4 的职责与终稿格式。
- 不解决误报为主的问题（仍由阶段 3 质检约束；合并阶段禁止裁证据）。

---

## 已确认决策（brainstorming）

| 项 | 选择 |
|----|------|
| 主要痛点 | **漏报** |
| 阶段 2 执行方式 | **三 agent 并行** |
| 合并去重 | **主编排**（不委派 merger） |
| 空结果可追溯 | 每 agent **必须**输出「扫描覆盖说明」 |
| 实现形态 | **方案 A**：单 `SKILL.md` 内委派三个角色 |

---

## 总体流程

```text
0. 读取 PR/diff/commit（主编排）
1. 变更意图分析（1 个 agent，不变）
2. 代码缺陷扫描（新）
   2a. 变更代码本身审查     ─┐
   2b. 变更周边影响审查     ─┼─ 并行
   2c. 目的与兼容性审查     ─┘
   → 主编排：合并候选缺陷 + 覆盖说明门禁
3. 缺陷质检（1 个 agent，规则微调）
4. 报告拼装（主编排，不变）
```

### 阶段 2 并行输入（三 agent 相同）

- diff 原文（或 PR 变更文件 diff）
- 阶段 1「变更意图分析」Markdown **全文**
- 全局段落（从 SKILL 复述）：缺陷成立条件、缺陷性质、缺陷等级、候选缺陷输出格式、反证检查、执行约束 1–7

### 阶段 2 主编排职责（新增）

1. **并行**发起 2a / 2b / 2c（`Task`，三条独立 sub-agent）。
2. 收集三份输出，将 `### 候选缺陷` 重编号（合并前保留来源前缀 `2a-`/`2b-`/`2c-` 便于追溯）。
3. **合并去重**（见 §合并规则）；**禁止**以证据不足为由删除（交给阶段 3）。
4. **覆盖说明门禁**：任一份「扫描覆盖说明」存在应扫未扫且无合理 skip → 主编排轻量补查，或**再委派该 agent 一次**（同一 diff + 意图分析 + 指出漏项）。
5. 将合并后的候选缺陷 Markdown 交给阶段 3。

## Specialist agent 规格

### 2a — 变更代码本身审查

**聚焦：** 变更 hunk 及**所在函数的完整实现**（必须 Read 含变更的完整函数体，禁止只看 diff 片段）。

**必扫 checklist（须全部出现在「扫描覆盖说明」勾选列表中）：**

- 语言/运行时：panic/recover、nil、zero/default 误用、OOM 风险、资源泄漏、并发竞态、错误处理遗漏
- 安全：认证/授权、输入校验、敏感信息、注入、路径、命令执行、越权
- 边界：nil/empty/zero/default、重复/缺失/非法值、超时/重试、并发、部分失败

**不做：** 全仓上游入口追踪（2b）；PR 叙述是否达成（2c）。

**Read 预算：** Read ≤35；Grep ≤12；Glob ≤8。

---

### 2b — 变更周边影响审查

**聚焦：** 以变更符号为锚的调用链与同类对比。

**必扫 checklist：**

- 上游：从可达入口到变更点；调用方参数/状态/错误处理/并发语义
- 下游：从变更点到关键下游；新参数/状态/错误值是否破坏假设
- 兄弟/同类：同文件、同包、同模式实现对比
- 残留：仅当阶段 1 `变更性质` 含 **bugfix** 时，Grep 同根因/同模式残留

**不做：** 变更函数内纯局部问题（2a），除非由链路透传导致。

**Read 预算：** Read ≤40；Grep ≤15。

---

### 2c — 目的与兼容性审查

**聚焦：** 声称 vs 实现；对外契约与升级。

**必扫 checklist：**

- 意图：对照 commit message、PR 标题/描述、comments——是否修好、是否部分路径、逻辑是否不可达
- 兼容性：API、配置、schema/CRD、默认值、数据迁移、滚动升级、回滚

**不做：** 全仓调用链深挖（2b）；泛化语言层扫描（2a）。

**Read 预算：** Read ≤30；Grep ≤10。

---

### 统一输出格式

```markdown
## 候选缺陷列表

### 候选缺陷 2a-1：标题

- 缺陷的性质：
- 缺陷等级：
- 相关代码证据：
- 缺陷的触发条件：
- 代码缺陷解读：
- 造成的代码后果和业务功能后果：
- 反证检查：
- 建议解决方案：

（若无：写「本 agent 范围内未发现满足成立条件的候选缺陷。」）

## 扫描覆盖说明

- 已检查的变更文件：
- 已覆盖的维度：（逐项勾选，与 agent checklist 一致）
- 未深入的原因：（无则写「无」）
- 结论：（一句总结）
```

候选缺陷 ID 前缀：`2a-` / `2b-` / `2c-`；主编排合并后改为终稿序号 `1..N`。

## 主编排合并规则

| 规则 | 说明 |
|------|------|
| 只合并、不裁证据 | 相同根因合并为一条，保留最全证据；不得以「可能性低」删除 |
| 跨 agent 重复 | 同一落点 + 同一根因 → 合并；不同根因 → **保留多条** |
| 覆盖门禁 | 覆盖说明中维度标「未检查」且无 skip 理由 → 补查或再委派该 agent 一次 |
| 三份皆零缺陷 | 允许进入质检；质检不得因「均未报」跳过逐条反证核查 |

**合理 skip 示例（须在覆盖说明写明）：** 纯 `docs`/`*.md` 变更跳过上下游；纯格式化无行为变化等。

---

## 阶段 3 质检（增量规则）

在现有质检规则上**追加**：

1. **删除门槛不变**：仅反证后证据不足可删；不得因「其他 agent 未报」而删。
2. **可补充不可删**：模糊但成立 → 补证据/触发条件；禁止仅润色后删除。
3. **核实范围**：不重新全量审计；允许对单条候选 Read 锚点函数核实反证（建议 ≤10 Read/条）。

其余：模糊词黑名单、P3 忽略、成立/不成立处理规则 **不变**。

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| Token/时延上升（3 并行） | 各 agent Read 预算上限；职责边界减少重复深读 |
| 跨 agent 重复 finding | 主编排合并规则 + 不同根因保留多条 |
| 主编排合并时误删 | 合并阶段禁止裁证据；质检规则加固 |
| SKILL.md 过长 | v2 再将 2a/2b/2c prompt 抽到 `agents/*.md` |
