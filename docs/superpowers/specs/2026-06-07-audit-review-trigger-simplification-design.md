# 设计文档：audit/review — 触发条件精简与非触发场景（T3 v3 / T4）

- 日期：2026-06-07
- 状态：approved（用户审阅通过，待 implementation plan）
- 父文档：[`2026-06-07-audit-review-evidence-provenance-design.md`](2026-06-07-audit-review-evidence-provenance-design.md)
- 背景：T1 v2 + T3 v2 落地后，实际输出中「顶层逻辑条件」与「参考触发场景」大量重复（同一配置/输入在抽象条件与具象场景中各写一遍）。用户侧只需一次说清：配了什么、外部 I/O 是什么、应用层发生什么。另需可选反例帮助读者理解「什么情况下不会出问题」。

## 1. 问题陈述

以 vLLM 弹性 EP 扩展缺陷为例，当前典型冗余：

| 顶层逻辑条件 | 参考触发场景 |
| --- | --- |
| 条件1：env `VLLM_RAY_DP_PLACEMENT_NODE_IPS` 设为受限 IP 列表 | 用户配置：同一 env = `"10.0.0.1,10.0.0.2"` |
| 条件2：触发弹性 EP 扩展（`scale_up_elastic_ep`） | 外部输入输出：弹性扩展 API + `new_data_parallel_size` |
| 映射条件：条件1 + 条件2 | （重复表达 AND 关系） |

**读者目标：** 在「缺陷的触发条件」中直接获得可操作的复现场景，而非先读抽象布尔前提再读具象 vignette。

## 2. Brainstorming 决策摘要

| 决策点 | 选择 |
| --- | --- |
| 顶层逻辑条件 | **删除**（含条件编号、`映射条件`） |
| 参考触发场景 | **保留**，作为触发条件的**唯一必填**块 |
| 多触发路径 | 多条参考场景，各自自洽完整（方案 A） |
| 非触发场景 | **新增**，**可选**（方案 C：有则写，无则省略，不影响 finding 成立） |
| 非触发场景数量 | 每条缺陷最多 1 条 |
| 范围 | 仅 `plugins/audit/skills/review/SKILL.md`；investigate-issue R21 不在本次范围 |

## 3. 终稿字段结构

```markdown
- 缺陷的触发条件：
  - **参考触发场景**（≥1 条；T3 v3）：
    - **场景1**（来源：code_synth | pr_context | hybrid）
      - **用户配置**：
        - {配置面} `{键}` = `{取值}`
          - 定义/注册：path:line
          - 读取/生效：path:line
        - 或：无
      - **外部输入输出**：
        - {I/O 类型} {请求标识或来源服务} …
          - 关键参数/字段：…
          - 定义/注册：path:line
          - 传入软件：path:line
        - 或：无（纯用户配置/内部运行时触发）
      - **应用层行为**：用户可理解的操作与结果（非函数名罗列）
    - **场景2**（…）：…
  - **非触发场景**（可选；最多 1 条）：
    - **对比说明**：相对哪条参考场景、哪项配置或外部 I/O 不同
    - **用户配置** / **外部输入输出**（只写差异项；结构同参考场景）
    - **应用层行为**：此情况下正常发生什么
```

**删除字段：** `顶层逻辑条件`、`映射条件`、`条件1/条件2` 编号体系。

## 4. 三块职责边界

| 块 | 回答的问题 |
| --- | --- |
| **R1 可达性证据** | 用户可见入口 → 完整调用链 → 防护为何未生效 |
| **参考触发场景（T3 v3）** | 用户侧如何复现：配置 + 外部 I/O + 可观察行为 |
| **非触发场景（T4，可选）** | 举一个反例：何种配置/输入下**不会**出问题 |
| **反证检查**（已有，不变） | 代码里 validation/兜底/兄弟路径为何没挡住 |

非触发场景面向**用户可操作的对比**；反证检查面向**代码路径**。二者互补，不得把反证检查里的代码兜底复述为非触发场景。

## 5. T3 v3 — 参考触发场景（唯一必填触发块）

相对 T3 v2 的变更：

- **删除**对顶层逻辑条件的依赖；**删除** `映射条件` 字段
- 保留：场景来源、用户配置、外部输入输出、应用层行为（结构化出处要求不变）
- 多路径：写多条场景，每条独立可读；同一路径只需一条，禁止为凑数拆分

### 5.1 用户配置 — 结构化条目（不变）

每个配置项一条：配置面、键或参数、取值、定义/注册 refs、读取/生效 refs。

### 5.2 外部输入输出 — 结构化条目（不变）

分两类：外部客户端输入、外部服务响应。每条条目含 I/O 类型、请求标识或来源服务、关键参数/字段与取值、定义/注册 refs、传入软件 refs。

无外部 I/O 时：

```markdown
- **外部输入输出**：无（纯用户配置/内部运行时触发）
```

内部模块间函数调用只写 R1-b，**不得**写入外部输入输出。

### 5.3 禁止词（不变）

大概、可能、也许、似乎、潜在、有一定概率、在某些情况下、未来、将来、若以后、一旦升级后（无当前代码依据时）、无主体操作（用户执行某操作后、配置不当时）。

## 6. T4 — 非触发场景（可选）

### 6.1 何时写

- **写**：能从代码/配置明确推断出与某条参考场景仅差一两项的反例时
- **省略**：纯内部竞态、无用户可见开关、无法构造有意义反例时

### 6.2 编写要求

- 每条缺陷**最多 1 条**；**可不写**
- 若写了，须标明**对比说明**（相对哪条参考场景、差异是什么）
- 只写与参考场景的**差异项**（用户配置和/或外部输入输出）
- 差异项须有配置面/I/O 类型 + refs（与参考场景同等要求）
- 须有一句**应用层行为**说明此情况下正常发生什么
- 适用与参考场景相同的禁止词表

### 6.3 与反证检查的区分

| 反证检查 | 非触发场景 |
| --- | --- |
| 上游是否已过滤 | 用户未做某配置时是否正常 |
| validation 是否已处理 | 用户改用安全取值时是否正常 |
| 下游是否已兜底 | 用户未调用某 API 时是否正常 |
| 代码路径分析 | 用户侧可复现的反面例子 |

## 8. Agent 工作流（阶段 2 输出前）

原步骤「合成场景 + 映射逻辑条件编号」改为：

1. 从 diff/锚点提取配置键、默认值、schema、常量、外部 API 定义（Grep/Read）
2. 从阶段 1 变更声称提取 `pr_context`（若有）
3. 合成 ≥1 条自洽参考场景（用户配置 + 外部 I/O + 应用层行为 + refs）
4. **若**能从代码推断清晰反例 → 写 1 条非触发场景；否则省略
5. 构建 E1 变更点证据、R1 可达性证据（不变）
6. 自检：参考场景无禁止词；无内部调用误作外部 I/O；非触发场景（若有）有对比说明 + refs

## 9. 内容归属表（更新）

| 内容 | 归属 |
| --- | --- |
| env / CLI / YAML / CR 用户配置 | 参考触发场景 → 用户配置 |
| 外部 HTTP/CLI/gRPC 请求 | 参考触发场景 → 外部输入输出 → 客户端输入 |
| Ray/K8s/第三方 API 响应 | 参考触发场景 → 外部输入输出 → 外部服务响应 |
| 安全/反面配置取值 | 非触发场景 → 用户配置 / 外部输入输出（可选） |
| 内部模块间函数调用 | R1-b 完整调用链 |
| 调用链、函数分支、`ptr==nil` 落点 | R1-b / R1-c |
| validation/兜底为何未生效 | 反证检查 |
| 用户可观察的错误结果 | 应用层行为 + 造成的后果 |

## 10. 正反例（vLLM 弹性 EP）

### 10.1 好例子

```markdown
- 缺陷的触发条件：
  - **参考触发场景**：
    - **场景1**（来源：code_synth）
      - **用户配置**：
        - 环境变量 `VLLM_RAY_DP_PLACEMENT_NODE_IPS` = `"10.0.0.1,10.0.0.2"`
          - 定义/注册：envs.py:160
          - 读取/生效：utils.py:530
      - **外部输入输出**：
        - 外部客户端输入：弹性扩展 API
          - 关键参数：`new_data_parallel_size` = …
          - 传入软件：utils.py:830
      - **应用层行为**：扩展时 placement group 可能落在不在允许列表中的节点上
  - **非触发场景**：
    - **对比说明**：相对场景1，未设置节点 IP 限制
    - **用户配置**：
      - 环境变量 `VLLM_RAY_DP_PLACEMENT_NODE_IPS` 未设置
        - 定义/注册：envs.py:160
        - 读取/生效：utils.py:530（使用默认全节点 placement）
    - **应用层行为**：弹性扩展可在集群可用节点上正常创建 placement group
```

### 10.2 坏例子

| 写法 | 问题 |
| --- | --- |
| 顶层逻辑条件 + 参考场景重复写同一 env | T1/T3 冗余（v2 问题，v3 已删除 T1） |
| 非触发场景写「上游 validation 会检查 IP」 | 混淆反证检查与非触发场景 |
| 非触发场景无对比说明 | `non_trigger_no_contrast` |
| 参考场景仍含 `映射条件：条件1 + 条件2` | 已删除字段 |
| 内部调用 `scale_up_elastic_ep` 写入外部输入输出且无 I/O 类型 | `trigger_scenario_internal_as_external` |
