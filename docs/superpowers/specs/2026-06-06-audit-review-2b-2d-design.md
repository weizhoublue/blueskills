# audit review 阶段 2b 上下游扫描重构 & 2d 同类残留拆分设计

**日期：** 2026-06-06  
**状态：** 已实施  
**插件：** `plugins/audit`（skill：`review`）

---

## 背景

`plugins/audit/skills/review/SKILL.md` 阶段 2 已拆为 2a / 2b / 2c 三个并行 specialist。实践中 **2b — 变更周边影响审查** 的上下游指引过于含糊：

- 「从软件顶层可达入口一直逐层追踪到变更点」范围不可操作，agent 常在「顶层入口」上纠结或浅尝辄止；
- 「关键下游」无定义，发现与判断混在一起；
- 「同类残留」与上下游、兄弟对比同属 2b，职责混杂，且与 2c「修复完整性」边界不清。

另：各 agent 的 **Read 预算** 在实践中易被当作提前止损借口，与「扫全清单」目标冲突。

---

## 目标

1. **2b 可执行化**：将上下游审查拆为「一层扫描建清单 → 逐项核实」，多锚点合并去重，核实中按需再扩一层并记录。
2. **残留专项化**：将全仓「同类残留」从 2b 拆出为 **2d**，仅在阶段 1 判定为 **bugfix** 时委派。
3. **职责边界清晰**：修复完整性（本 PR 是否修好声称问题）留在 **2c**；2d 只做全仓同模式残留。
4. **去除 Read 上限**：删除 2a/2b/2c 及阶段 3 中所有 Read 数量 cap；Grep/Glob 上限保留。
5. **保持输出契约**：候选缺陷格式、缺陷成立条件、P0–P2、最终报告与 `REVIEW_RESULT` 不变；仍由主编排合并 + 覆盖说明门禁。

---

## 非目标

- 不新增独立 merger sub-agent。
- 不改变阶段 1 / 阶段 4 职责与终稿格式。
- 不将 2c 的「修复完整性」迁入 2d。
- 不将 2b「兄弟/同类对比」（同文件、同包、同业务模式并行实现）迁入 2d。
- v1 仍只改 `SKILL.md`（及必要时 `plugin.json` 版本/描述）；不新增 `agents/*.md`。

---

## 已确认决策（brainstorming）

| 项 | 选择 |
|----|------|
| 多变更锚点 | **合并去重**：所有锚点的一层上下游汇总后统一核实 |
| 核实边界 | **按需扩展**：清单项核实中发现具体疑点时可再扩一层，须在覆盖说明记录 |
| 同类残留 | **拆为 2d**，仅 bugfix 时委派；2b 删除残留项 |
| 修复完整性 | **留在 2c**（本 PR 变更范围内是否达成声称修复） |
| 兄弟/同类对比 | **留在 2b**（并行实现对比，非全仓 Grep 残留） |
| Read 预算 | **全文删除**（2a/2b/2c/阶段 3） |

## 2b — 变更周边影响审查（重写）

**角色：** 变更周边影响审查员  
**聚焦：** 以变更符号为锚的**一层**调用关系 + 兄弟/同类并行实现对比。

### 必扫流程（须按序执行）

#### 阶段 A — 锚点与一层扫描（只建清单，不做缺陷判断）

1. **提取锚点**：从 diff 列出所有变更的函数/方法（含新增、修改、删除体；纯类型/常量变更则锚点为所在作用域最近的可调用单元）。
2. **对每个锚点扫描一层**：
   - **上游（callers）**：在同仓库内找**直接调用**该锚点的函数/方法（`Grep` 符号引用 + 必要时 `Read` 调用点上下文）。
   - **下游（callees）**：`Read` 锚点**完整函数体**，列出其**直接调用**的函数/方法（不猜测间接调用链）。
3. **合并去重**：跨锚点汇总，生成 **一层上下游清单**（写入「扫描覆盖说明」，必填）：

```markdown
### 一层上下游清单

| 方向 | 符号 | 文件 | 关联锚点 | 发现方式 |
|------|------|------|----------|----------|
| 上游 | callerFn | path/to/a.go | anchorFn | grep |
| 下游 | calleeFn | path/to/b.go | anchorFn | read body |
```

- 无直接 caller：行内注明 `无直接 caller（可能为入口/回调/仅测试引用）`。
- 无直接 callee：注明 `无直接 callee（叶子/内联逻辑）`。

**阶段 A 禁止**：输出缺陷结论、使用「可能有问题」等判断性表述、无清单项锚定地追第二层。

#### 阶段 B — 清单逐项核实

对清单**每一行**在覆盖说明中勾选核实（`- [x]`）：

| 方向 | 核实项 |
|------|--------|
| 上游 caller | 传入参数类型/个数/语义；前置状态；错误处理；并发/重入；变更后调用是否仍合法 |
| 下游 callee | 新参数/返回值/错误码；共享状态；下游对 nil/空/默认值假设是否仍成立 |

**按需扩展**（仅当核实中发现**具体疑点**，如参数透传未校验、新错误码未处理）：

- 可向该方向**再扩一层** caller 或 callee；
- 须在覆盖说明追加：**扩展项 | 因何疑点 | 扩到哪一层 | 结论**；
- **禁止**无清单项锚定的全仓深搜。

#### 阶段 C — 兄弟/同类对比（与 A/B 解耦）

- 同文件、同包、同业务模式的**并行实现**对比（非全仓 Grep 残留，后者属 2d）。

### 删除的旧表述

- 「从软件顶层可达入口一直逐层追踪到变更点」
- 「从变更点到关键下游」
- checklist 中的「残留」整条（迁至 2d）
- `Read ≤40` 及 2b 段 Read 预算行

### 不做

- 变更函数内纯局部问题（2a），除非由链路透传导致；
- 全仓同类残留 Grep（2d）；
- 本 PR 是否修好声称问题（2c）。

### 2b 覆盖说明门禁字段（主编排检查）

- `- [ ] 一层上下游清单已生成且覆盖全部锚点`
- `- [ ] 清单每一行已核实或注明 skip 原因`
- `- [ ] 扩展项（若有）已记录疑点与结论`
- `- [ ] 兄弟/同类对比已执行或注明 skip 原因`

### 输出

统一输出格式；候选缺陷 ID 前缀 `2b-`。  
**Grep 预算保留：** `Grep ≤15`（无 Read 上限）。

## 2d — 同类残留审查（新增）

**角色：** 同类残留审查员  
**触发：** 仅当阶段 1「变更性质」含 **bugfix**；否则主编排不委派。

**聚焦：** 当前 PR 修了一处，仓库中是否仍有**相同根因或高度相似模式**的残留（缺陷性质第 3 类：「仓库中其他代码的同类残留缺陷」）。

### 必扫流程（须按序执行）

#### 阶段 A — 残留模式提取（只建清单）

1. 从 diff + 阶段 1「变更声称要解决的问题」提取本次修复的**根因模式**（关键字、条件结构、错误处理缺口、API 误用方式等）。
2. 生成 1～3 条可 `Grep` 的**搜索模式**（写入覆盖说明；禁止空泛描述如「类似逻辑」）。
3. 全工程 `Grep`，产出 **同类残留候选清单**（去重）：

```markdown
### 同类残留候选清单

| 文件 | 函数/位置 | 匹配模式 | 与本次修复相似点 | 是否已读上下文 |
|------|-----------|----------|------------------|----------------|
```

#### 阶段 B — 逐项核实

对清单每一行 `Read` 上下文，判断是否满足共享「缺陷成立条件」中的**真实残留缺陷**。  
若该处已有 validation/兜底、或并非同根因 → 不输出。

### 不做

- 上下游调用链（2b）；
- 本 PR 变更范围内是否修好声称问题（2c）；
- 变更函数内局部缺陷（2a）。

### 输出

统一输出格式；候选缺陷 ID 前缀 `2d-`。  
建议 **Grep 预算：** `Grep ≤20`（可随实现微调；无 Read 上限）。

删除 `Read ≤30` 行；**Grep 预算保留：** `Grep ≤10`。

## 2a — 变更代码本身审查（仅预算调整）

## 主编排合并（更新）

1. **收集**：保留来源前缀 `2a-` / `2b-` / `2c-` / `2d-`（有则并）直至合并完成。
2. **合并去重**：
   - 相同根因 + 相同落点 → 合并为一条，保留最全证据；
   - 不同根因 → 保留多条；
   - **2b 兄弟对比** 与 **2d 残留** 若命中同一代码落点 → 合并为一条；
   - **禁止**以证据不足删除（交给阶段 3）。
3. **覆盖说明门禁**：
   - 已委派 agent 的 checklist 须全部勾选或合理 skip；
   - **2b**：须含完整一层上下游清单及逐项核实记录；
   - **2d**（已委派）：须含残留候选清单及逐项核实记录；
   - 未委派 2d（非 bugfix）→ 不检查 2d 覆盖说明。
4. **重编号** → 交给阶段 3。

### SKILL 文案需同步更新的位置

- 文首编排描述：阶段 2 为「2a/2b/2c（+ 条件 2d）」；
- §并行委派表格：增加 2d 行；
- 「三 agent 统一输出格式」→ 改为「各 agent 统一输出格式」（含 2d）；
- 阶段 1 限制：不得暗示 2d 可因「非 bugfix」由 2b 代劳（残留已迁出）。

## 验收标准

1. bugfix PR：主编排发起 2a+2b+2c+2d 四次并行 Task；2b 覆盖说明含完整一层上下游清单；2d 含残留候选清单。
2. 非 bugfix PR：仅三次 Task；2b 无残留 checklist；无 2d 输出。
3. SKILL 全文无 `Read ≤` 或「Read 预算」表述。
4. 2c 意图项明确不包含全仓残留；2d 明确不包含修复完整性。
5. 候选缺陷格式、最终报告、`REVIEW_RESULT` 与现网一致。

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 一层扫描漏掉跨文件间接影响 | 阶段 B 按需扩展 + 2a 函数体内扫描 |
| 2d Grep 噪声大 | 强制先写 1～3 条具体搜索模式；阶段 B 必读上下文 |
| 四次并行 token 增加 | 仅 bugfix 触发 2d；非 bugfix 路径不变 |
