

我想写一个新的 Plugin ，名为 investigate-issue ，那这个 plugin 呢，它具备一个名为 investigate  skill，这个 skill 呢可能有几个 agent 来组成。 


这个 plugin 提供的这套 skill 呢，主要是针对开源项目的某个问题进行深入地分析，给出一个报告。 

应该给出如下四个小节的覆盖。 

- **问题描述** 
这个小节主要是把用户说的这个问题进行全面的解释。这个解释要包含几个方面：
1. 代码函数级别的解释
2. 问题的解释
这个解释要更加深入，分析到前后剩下所有的调用链，告诉说这个是这个应用本身的某个 config 配置或者是哪个环境变量等等，然后触发，会进入这样的一个分支，进入这样一个函数，是要把整个调用链程度讲清楚。


然后，另外一个层面的这个解释，就是这个代码背后的这个业务级别的这种解释。它不是讲的是代码级别的，它讲的是这个问题背后的这个业务场景到底是什么样的一个上下游的逻辑关系。 


然后还有一个层面呢，就要讲清楚这个缺陷的代码分支为什么在同样的代码的其他的兄弟分支或者说在其他相同的业务功能模块中，它们为什么没有出现这个问题，或者说它们是不是也存在这个问题？为什么这个问题只出现于大家病症上，所以要讲清楚这个比较。 

- **问题后果** 
这个小节主要讲清楚两个层面：
1. 代码层面：这个缺陷会导致业务工作不正常，进入什么样的错误分支会引发代码错，数据丢失也好，竞争也好，panic 也好。

另一个层面就是要讲清楚它导致用户层面的影响以及功能上的影响：它是否导致某个业务级别的功能不正常了，或者是性能下降了，或者是 panic 了。这个是实际的影响，而不是代码层面的讲解分析。 


- **触发条件** 

要讲清楚当前这个问题的触发条件。就是说，一方面它要讲清楚用户配置了什么，或者说这个软件输入了什么，然后产生了什么样的函数调用链之后就会触发这个问题。 


- **背景知识** 
希望把这个出问题的这个代码对应的模块和它所对应的整个软件的功能，是功能层面，它相关的背景知识要讲清楚。 




我希望在实现这个报告的过程中，可能有多个 agent 分别去协同分工，分别完成报告的不同部分。

可能先要有一个信息搜集员，针对用户给的这些问题进行信息收集。 

然后，可能要有一个代码析员 , 针对搜集的信息，然后根据当前录下的代码进行分析。 
然后，这个代码分析员最重要的原则就是不能凭空想象，必须所有的一些依据都要在代码中找到。否则就不能作为信息来呈现。 

最重要的一定要有一个审计的 agent，然后他能够对这些报告的内容提出一些优化的建议。 
- 这些建议包括了这个输出的这些文字本身的因果关系，对不对？而且它的这个因果关系的层级是不是不够充分？也许它只停留在很表面，并没有把这个问题的更多层次的上下游的因果关系表述清楚。所以它应该要让生成报告这个 agent 吩咐这些层级关系的、推敲跟因果关系的表述明。 
- 那自己这个报告的一些表述，它是不是过于简短了？然后里面没有充分解释一些背景知识，所以这个报告其实对于小白是看不懂的。 
- 尤其是针对这个问题描述这个环节，他应该要让这个报告中完全体现出更深层级的逻辑关系，甚至是应用配置入口或者是应用输入输出级别触发了上下游代码的逻辑关系。一定要让报告中体现出来，而避免这个问题的分析只限于局部。所以，有时候其实就不知道所谓的局部是整个软件的哪一个部分。 


所以我希望让这个报告生成的 agent 跟这个审核的 agent 之间能够有最多达五轮的双方讨论过程，然后来最终不断地优化、丰富整个报告的输出。
